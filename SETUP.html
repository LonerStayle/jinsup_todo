Solo Todo — 첫 실행 SETUP 가이드

앱 코드 외부에서 한 번 셋업이 필요한 항목. 위에서 아래로 순서대로 진행해 주십시오. 체크박스를 하나씩 채우면서 진행하시면 빠짐없이 처리됩니다.

1 환경변수 .env.local

저장소 루트의 .env.example 을 .env.local 로 복사 후 4 키를 채우십시오. .env.local 은 .gitignore 에 포함되어 있어 커밋되지 않습니다.

키발급처비고
SUPABASE_URL Supabase Dashboard → Project Settings → API 예: https://abc.supabase.co
SUPABASE_ANON_KEY 동일 JWT (eyJ…)
GOOGLE_OAUTH_CLIENT_ID_DESKTOP Google Cloud Console → APIs & Services → Credentials Type: Desktop. macOS 빌드용
GOOGLE_OAUTH_CLIENT_ID_ANDROID 동일 Type: Android, package + SHA-1 fingerprint 필요

실행 시 주입

flutter run --dart-define-from-file=.env.local
flutter build macos --release --dart-define-from-file=.env.local
flutter build apk --release --dart-define-from-file=.env.local
  • Supabase URL + Anon Key 입력
  • Google OAuth Desktop client id 입력 (Calendar 안 쓸 거면 비워둬도 OK)
  • Google OAuth Android client id 입력

2 Supabase 프로젝트 + RLS

  1. Supabase Dashboard 에서 새 프로젝트 생성 (region: ap-northeast-2 서울 권장).
  2. 좌측 메뉴 Authentication → Providers → Email 활성화. 매직링크가 아니라 OTP 코드 (6~10자리) 흐름으로 사용 — SignInScreen 이 코드 입력 UI 제공.
  3. Database → SQL Editor 에서 아래 SQL 실행. 저장소에도 파일로 있습니다:
    • 신규 셋업: supabase/schema.sql
    • 이전 버전에서 옮길 때: supabase/migrate.sql
-- 별도 schema 'solo_todo' 안에 격리. 다른 앱이 같은 프로젝트의 public 을 쓰더라도 영향 X.
create schema if not exists solo_todo;

-- Supabase 의 PostgREST + Realtime 이 사용하는 role 에 schema 사용 권한 부여.
grant usage on schema solo_todo to anon, authenticated, service_role;
alter default privileges in schema solo_todo
  grant all on tables to anon, authenticated, service_role;
alter default privileges in schema solo_todo
  grant all on sequences to anon, authenticated, service_role;

-- todos 테이블
create table if not exists solo_todo.todos (
  id                text primary key,
  user_id           uuid references auth.users(id) on delete cascade not null,
  title             text not null,
  category          text not null,
  due_at            timestamptz,
  done_at           timestamptz,
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now(),
  calendar_event_id text
);

-- 이미 만든 테이블에도 grant (default privileges 는 새 객체에만 적용되므로 명시)
grant all on solo_todo.todos to anon, authenticated, service_role;

-- RLS — 본인 row 만. drop-then-create 로 재실행 안전.
alter table solo_todo.todos enable row level security;
drop policy if exists "user_read_own"   on solo_todo.todos;
drop policy if exists "user_insert_own" on solo_todo.todos;
drop policy if exists "user_update_own" on solo_todo.todos;
drop policy if exists "user_delete_own" on solo_todo.todos;
create policy "user_read_own"   on solo_todo.todos for select using (auth.uid() = user_id);
create policy "user_insert_own" on solo_todo.todos for insert with check (auth.uid() = user_id);
create policy "user_update_own" on solo_todo.todos for update using (auth.uid() = user_id);
create policy "user_delete_own" on solo_todo.todos for delete using (auth.uid() = user_id);

-- 동기화 정렬용 인덱스
create index if not exists todos_user_updated on solo_todo.todos (user_id, updated_at desc);

-- Realtime publication 에 추가. 이미 멤버면 42710 에러나므로 조건부.
do $$
begin
  if not exists (
    select 1 from pg_publication_tables
    where pubname = 'supabase_realtime'
      and schemaname = 'solo_todo' and tablename = 'todos'
  ) then
    alter publication supabase_realtime add table solo_todo.todos;
  end if;
end $$;
Supabase Dashboard → Settings → API → Exposed schemas 에 solo_todo 를 반드시 추가해 주십시오. (default 는 public, graphql_public 만이라 새 schema 는 명시해 줘야 PostgREST 가 인식합니다.)

이전 버전에서 public 에 테이블을 만들었었다면 (정리)

예전에 public.todos 또는 public.solo_todo_todos 를 만든 적이 있다면 아래 정리 SQL 을 위 메인 SQL 보다 먼저 실행해 옛 테이블을 삭제하십시오. 초기 셋업이라 실데이터가 없다는 전제 — 데이터 보존이 필요하면 drop 대신 alter table … set schema solo_todo 로 옮겨야 합니다. (파일: supabase/migrate.sql)

-- 1) publication 에서 먼저 제거 (ALTER PUBLICATION 은 IF EXISTS 미지원 → 조건부)
do $$
begin
  if exists (select 1 from pg_publication_tables
             where pubname='supabase_realtime' and schemaname='public' and tablename='todos') then
    alter publication supabase_realtime drop table public.todos;
  end if;
  if exists (select 1 from pg_publication_tables
             where pubname='supabase_realtime' and schemaname='public' and tablename='solo_todo_todos') then
    alter publication supabase_realtime drop table public.solo_todo_todos;
  end if;
end $$;

-- 2) 옛 테이블 삭제 (policy / index 함께 cascade). 없으면 no-op.
drop table if exists public.todos cascade;
drop table if exists public.solo_todo_todos cascade;
  1. Settings → API → Exposed schemas 에 solo_todo 추가 (default public, graphql_public 옆에 콤마 + solo_todo).
  2. Authentication → URL Configuration → Redirect URLs 에 매직링크 콜백 URL 추가:
    • macOS: io.supabase.solo_todo://login-callback
    • Android: com.goldenplanet.solo_todo://login-callback
  • 프로젝트 생성
  • Email Magic Link 활성화
  • SQL 실행 (schema + todos 테이블 + RLS + Realtime add)
  • Exposed schemas 에 solo_todo 추가
  • Redirect URLs 두 개 등록

v1.0 → v1.1 마이그레이션 (트리 / 메모 모델)

이미 v1.0 시점에 위 SQL 을 실행한 프로젝트는 todos 테이블에 신규 3 컬럼이 빠진 상태. SQL Editor 에서 한 번 실행 (idempotent):

alter table solo_todo.todos add column if not exists parent_id  text;
alter table solo_todo.todos add column if not exists type       text not null default 'task';
alter table solo_todo.todos add column if not exists sort_order integer not null default 0;

신규 셋업 (v1.1 schema.sql 그대로 실행) 은 처음부터 포함되어 있어 이 ALTER 불필요.

v1.1 → v1.2 마이그레이션 (카테고리 동적 + 상세 메모)

v1.2 는 카테고리를 DB row 로 저장 (추가 / 삭제 가능) + todos 에 상세 메모(description) 추가. 가장 간단한 방법은 supabase/schema.sql 전체를 SQL Editor 에 다시 붙여넣고 실행하는 것입니다 (모든 구문이 idempotent — categories 테이블 / RLS / 인덱스 / publication / description 컬럼이 없으면 생성, 있으면 그대로). 또는 아래 핵심 구문만 실행:

-- categories 테이블 (사용자가 추가/삭제하는 카테고리)
create table if not exists solo_todo.categories (
  id              text primary key,
  user_id         uuid references auth.users(id) on delete cascade not null,
  label           text not null,
  icon_code_point integer not null,
  color_value     integer not null,
  sort_order      integer not null default 0,
  is_builtin      boolean not null default false,
  created_at      timestamptz not null default now()
);
grant all on solo_todo.categories to anon, authenticated, service_role;
alter table solo_todo.categories enable row level security;
-- RLS 4 정책 + 인덱스 + Realtime publication 은 schema.sql 의 8~10 섹션 참고.

-- todos 상세 메모 컬럼
alter table solo_todo.todos add column if not exists description text;

5 기본 카테고리(회사 할일 / 개인개발 / 일상 / 장기 목표 / 아이디어)는 앱이 첫 실행 시 로컬에 자동 seed 하고, 로그인하면 Supabase 로 동기화됩니다. 기존 todos 의 category 값(work 등)은 그대로 유지되어 호환됩니다.

3 Google Cloud (Calendar API)

  1. console.cloud.google.com 에서 새 프로젝트 생성.
  2. APIs & Services → Library → "Google Calendar API" Enable.
  3. OAuth consent screen 설정 (External + 본인 이메일).
  4. Credentials → + Create credentials → OAuth client ID:
    • Desktop app 타입 → client ID 발급 → GOOGLE_OAUTH_CLIENT_ID_DESKTOP 에 입력
    • Android 타입 → package: com.goldenplanet.solo_todo + SHA-1 fingerprint 
      cd android && ./gradlew signingReport
      출력 중 SHA1 값을 복사. GOOGLE_OAUTH_CLIENT_ID_ANDROID 에 입력.
  5. Scope 는 https://www.googleapis.com/auth/calendar.events 한 개. 코드가 자동 요청.
  • Calendar API Enable
  • OAuth consent screen 설정
  • Desktop client ID 발급 + .env.local 입력
  • Android client ID 발급 + .env.local 입력

6 OTP 인증 (이메일 코드)

본 앱은 매직링크가 아니라 OTP 코드 흐름을 사용합니다. 사용자는 이메일로 받은 6~10자리 코드를 SignInScreen 에 입력. URL scheme / deep link 등록 불필요.

Supabase 이메일 템플릿 — {{ .Token }} 노출

좌측 메뉴 Authentication → Email Templates 에서 두 템플릿을 OTP 표시용으로 수정:

  • Confirm signup: 본문에 {{ .Token }} 포함되도록 편집 (기본은 {{ .ConfirmationURL }} 만 있음).
  • Magic Link: 같은 방식 — {{ .Token }} 추가.

예: 당신의 인증 코드는 <strong>{{ .Token }}</strong> 입니다.

OTP 길이 (선택)

Supabase 의 OTP 길이는 프로젝트 설정에서 6~10 자리 가변. 앱은 6자리 이상이면 모두 수용 (입력 후 300ms idle 시 자동 verify).

  • Email Templates 의 Confirm signup 에 {{ .Token }} 노출
  • Email Templates 의 Magic Link 에 {{ .Token }} 노출
  • 테스트 — 빈 이메일로 "코드 받기" → 메일 도착 → 6~10자리 코드 입력 → 세션 생성

Supabase 대시보드
